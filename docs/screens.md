# Карта экранов

Документ описывает все экраны GUI и типичные пользовательские потоки.
Скриншоты не вложены (учебный проект), но компоновку легко воспроизвести
по описанию ниже.

## Окно логина

Поля: логин + пароль. Кнопка «Войти» (или `Enter`). При неудаче — красная подпись
с сообщением (`Неверный логин или пароль`, `Учётная запись заблокирована`,
`Слишком много неудачных попыток…`). После 5 неудач за 5 минут — лок-аут.

При успехе: окно скрывается, поднимается `MainWindow`, в `AppBus` эмитится
`user_logged_in(AuthContext)`. По «Выйти» происходит обратное.

## Справочники / Тренировки / Походы

Универсальная страница `EntityListPage`:

```
┌──────────────┬─────────────────────────────────────────────┐
│ Список сущн. │  CrudView для выбранной сущности            │
│ 220 px       │  [H1] Название (Туристы / Маршруты / …)     │
│ Физлица      │  [поиск…] [+ Добавить] [Изменить] [Удалить] │
│ Тренеры      │  ┌──────────────────────────────────────┐  │
│ Туристы      │  │ Таблица                              │  │
│ Секции       │  │  …                                   │  │
│ Группы       │  └──────────────────────────────────────┘  │
│ …            │  ← Назад | 1–25 из 137 | Вперёд →           │
└──────────────┴─────────────────────────────────────────────┘
```

Двойной клик по строке = «Изменить». Перед удалением — подтверждение. Если есть
зависимости — `ConflictError` ловится и показывается в красной подписи под
таблицей.

Формы редактирования (`FormDialog`) строятся из `FormField`-ов: текст/числа/
дата/bool/choice/relation (FK-комбобокс с подгрузкой связанных).
Обязательные поля отмечены `*`.

## SQL-консоль

```
┌────────────────────────────────────────────────────────────┐
│ [H1] SQL-консоль                                            │
│ [● Read-only] [⚠ Full (RW)]   режим: только чтение …       │
│                                История: [▾ — выбрать …]    │
│ ┌────────────────────────────────────────────────────────┐ │
│ │ -- введите SQL и нажмите «Выполнить» (Ctrl+Enter)      │ │
│ │ SELECT * FROM tourist;                                 │ │
│ └────────────────────────────────────────────────────────┘ │
│ [Выполнить] [Применить (COMMIT)] [Откатить] … [Экспорт CSV]│
│ status: 5 мс · Read-only: транзакция автоматически откачена│
│ ┌────────────────────────────────────────────────────────┐ │
│ │ Таблица результата                                     │ │
│ └────────────────────────────────────────────────────────┘ │
└────────────────────────────────────────────────────────────┘
```

- В **read-only** — кнопки Apply/Rollback скрыты, после каждого запуска
  транзакция сразу откатывается.
- В **full** — после `Выполнить` транзакция «висит»; нужно явно `COMMIT` или
  `ROLLBACK`. Смена режима с открытой транзакцией автоматически откатывает.

## Администрирование

`AdminPage` = `QTabWidget` с тремя вкладками.

### Пользователи

Таблица: `id / login / Активен / Создан / Последний вход`.
Toolbar: `+ Пользователь`, `Заблокировать`/`Разблокировать` (динамически),
`Сбросить пароль…`, `Роли…`.

- Создание — диалог с логином, паролем (с подтверждением длины) и multi-select
  ролями (чекбоксы, без системных).
- Назначение ролей — диалог-список со всеми ролями и галочками на текущих;
  при сохранении пересчитывается дельта и зовётся `assign_role` / `revoke_role`.
- Встроенного `admin` нельзя заблокировать и нельзя снять с него `superadmin`.

### Роли

Слева — список (системные помечены). Справа — чекбокс-матрица прав,
сгруппированных по entity (`tourist.*`, `route.*`, `admin.*`, `sql.*`,
`service.*`). Системная `superadmin` показана, но чекбоксы и кнопки disabled.

Кнопки: `+` создать, `✎` переименовать, `🗑` удалить, `Сохранить права`.

### Журнал входов

Таблица: `id / Время / Логин / ✓ или ✗ / user_id / IP`.
Фильтры: подстрока логина + результат (любой / успех / отказ).
Пагинация 100/страницу.

## Сервисный режим

Три карточки-действия + лог операций:

```
┌────────────────────────────────────────────────────────────┐
│ [H1] Сервисный режим                                       │
│ Эти операции необратимы и затрагивают всю базу…            │
│ ┌── Card: Очистить БД ─────────────────── [Очистить] ──┐   │
│ ┌── Card: Засеять демо-данными ────────── [Засеять]  ──┐   │
│ ┌── Card: Экспортировать дамп ────────── [Сохранить…] ──┐  │
│ [H2] Протокол операций                                    │
│ ┌──────────────────────────────────────────────────────┐   │
│ │ [14:23:11] Готов к работе.                           │   │
│ │ [14:23:42] ✓ Демо-данные засеяны: difficulty=6, …    │   │
│ └──────────────────────────────────────────────────────┘   │
└────────────────────────────────────────────────────────────┘
```

- **Очистка** — двойное подтверждение (`warning` + `critical`).
- **Посев** — идемпотентен: повторный вызов = no-op с уведомлением.
- **Экспорт** — `QFileDialog` с именем `tourist_club_YYYYMMDD_HHMMSS.sql`,
  затем `pg_dump` через subprocess.

## Типичные пользовательские потоки

1. **Первый запуск** (superadmin):
   admin → «Сервисный режим» → «Засеять» → видны данные везде.
2. **Демо запросов**:
   «Запросы по варианту» → выбрать 7 → задать тренера/период → «Выполнить» →
   «Экспорт CSV».
3. **Создание роли**:
   «Администрирование → Роли» → `+ Редактор` →
   отметить `tourist.read/create/update/delete` + `route.read` → Сохранить.
4. **Создание пользователя с ролью**:
   «Администрирование → Пользователи» → `+ Пользователь` → логин + пароль +
   галочка «Редактор» → Сохранить.
   Логаут → вход под новой учёткой → виден только нужный набор разделов.
5. **Произвольный SQL**:
   «SQL-консоль» в read-only → `SELECT ...` → видно результат, нельзя писать.
   Переключить в Full → `UPDATE ...` → видно «Транзакция открыта» →
   `Применить` / `Откатить`.
6. **Аудит**:
   неудачные входы видны в «Администрирование → Журнал входов»;
   после 5 неудач за 5 минут под одним логином дальнейшие отвергаются с
   `Слишком много неудачных попыток`.
