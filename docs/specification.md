# Спецификация: ИС туристического клуба

Документ фиксирует конкретные требования к проекту по курсу «Базы данных» (вариант 4.17).
Это контракт реализации: всё, что здесь не указано, в систему не входит.

## 1. Контекст

- **Предметная область:** туристический клуб (секции, группы, тренировки, походы, маршруты) — см. [task.md](task.md).
- **Пользователь системы:** сотрудники клуба (руководители секций, тренеры, администратор).
  Туристы в системе — это данные, а не пользователи.
- **Тип приложения:** desktop, single-user на каждой машине, подключается к общей БД.

## 2. Технологический стек (зафиксирован)

| Слой         | Решение                                       |
|--------------|-----------------------------------------------|
| Язык         | Python 3.12+                                  |
| UI           | PySide6 (Qt 6), кастомный QSS-движок тем     |
| ORM          | SQLAlchemy 2.0 (typed, declarative)           |
| Миграции     | Alembic                                       |
| СУБД         | PostgreSQL 16 (в Docker)                      |
| Просмотр БД  | Adminer (в том же docker-compose, порт 8080)  |
| Конфиг       | pydantic-settings + `.env`                    |
| Хеш паролей  | argon2-cffi                                   |
| Логирование  | стандартный `logging` + цветной форматтер     |
| Менеджер пак.| uv (lock + venv)                              |
| Качество     | Ruff (lint + format), mypy (strict для core)  |
| Тесты        | pytest + pytest-qt (минимальный smoke-набор)  |

**Обоснования:**
- PostgreSQL — взрослая SQL-СУБД с богатыми возможностями триггеров/проверок; Adminer даёт быстрый GUI «посмотреть, что внутри» без установки лишнего.
- SQLAlchemy 2.0 — типизированная ORM, проще, чем JDBC-стиль; Alembic закрывает миграции и историю схемы.
- pydantic-settings — типобезопасный конфиг из `.env`, легко документировать.

## 3. Доменная модель (сущности)

Базовый набор таблиц (детали полей уточняются на этапе ER-модели; здесь — обязательный минимум):

1. **Section** — секция (название, описание, head_id → Person).
2. **Group** — группа (название, section_id, trainer_id → Person, расписание тренировок раскрывается через `TrainingSession`).
3. **Person** — общий базовый человек (ФИО, пол, дата рождения).
4. **Tourist** — турист (наследует Person через FK 1—1, max_passed_difficulty_id → Difficulty).
5. **Athlete** — спортсмен (расширяет Tourist; разряд, специализация).
6. **Trainer** — тренер (расширяет Person; зарплата, специализация, дата приёма, section_id).
7. **SectionHead** — руководитель секции (расширяет Person; зарплата, дата приёма) — может быть и тренером в той же таблице через флаг, но для простоты выделяем отдельную сущность.
8. **TouristGroupMembership** — связь Tourist ↔ Group (даты вступления/ухода).
9. **TrainingSession** — тренировка (group_id, trainer_id, дата/время, длительность, место, тип занятий).
10. **Attendance** — посещаемость (training_session_id, tourist_id, present:bool).
11. **TrainerWorkload** — нагрузка тренера (trainer_id, вид занятий, период, часы). Может быть представлением (`VIEW`) над `TrainingSession`.
12. **Competition** — соревнование (название, дата, место, дисциплина).
13. **CompetitionParticipation** — участие (competition_id, person_id, результат).
14. **Route** — маршрут (название, длина_км, тип `hike|horse|water|mountain`).
15. **RoutePoint** — точка маршрута (route_id, порядковый_номер, название, координаты опционально).
16. **Difficulty** — категория сложности (название/код, минимальная длина, мин. дней — справочник).
17. **Trip** — поход (route_id, instructor_id → Person, дата_начала, дней, kind `planned|unplanned`, difficulty_id, parent_trip_id для перевода неплан.→план.).
18. **TripPlan** — план похода (trip_id, день, привалы, стоянки) — только для `planned`.
19. **TripDiary** — дневник (trip_id, день, текст) — только для `planned`.
20. **TripParticipant** — участник похода (trip_id, person_id, роль `participant|instructor`).

**Системные таблицы (ACL):**

21. **AppUser** — пользователь приложения (login, password_hash, is_active, person_id?).
22. **Role** — роль (имя, описание, is_system:bool).
23. **Permission** — справочник прав (код вида `tourist.read`, `competition.create`, `admin.users`, `sql.execute`, …) — наполняется фикстурой.
24. **RolePermission** — связь.
25. **UserRole** — связь.
26. **AuditLogin** — журнал входов (user_id, ts, ip/host, success).

## 4. Запросы по варианту

Все 13 запросов из [task.md](task.md) реализуются как отдельные экраны/формы:
форма с фильтрами слева, таблица результатов справа, в подвале — итог «Всего: N».
Внутри — параметризованные SQL/ORM-запросы (никаких конкатенаций строк).
Каждый запрос предоставляет экспорт в CSV.

## 5. Функциональные требования

### 5.1 Аутентификация
- Окно логина при старте; ввод логина и пароля; argon2id для хеша.
- Блокировка учётной записи после 5 неудачных попыток подряд (на 5 минут, через `AuditLogin`).
- Кнопка «выйти» возвращает к окну логина без перезапуска.

### 5.2 Авторизация (ACL — на оценку «5»)
- Проверка прав происходит **в сервисном слое** (не в UI) — UI лишь скрывает недоступные действия.
- Атрибут метода или декоратор `@require("tourist.delete")` бросает `PermissionDenied`.
- Встроенная роль **`superadmin`** (`is_system=True`) — обладает всеми правами, нередактируема, удалить нельзя.
- Встроенный пользователь `admin` создаётся миграцией с ролью `superadmin`; пароль задаётся через `.env` при первом запуске (или дефолт `admin`, с принудительной сменой).
- Суперадмин в админ-панели может:
  - создавать/редактировать/удалять роли (кроме системных);
  - назначать роли набор разрешений (чекбоксы);
  - создавать/блокировать пользователей, назначать им роли.
- Гранулярность прав — на уровне «сущность × операция» (`<entity>.<read|create|update|delete>`) плюс системные (`admin.users`, `admin.roles`, `sql.execute`, `service.testdata`).

### 5.3 CRUD
- Для каждой доменной сущности — общий шаблон экрана: таблица с пагинацией (page size настраивается), фильтр по тексту, кнопки `+`, `✎`, `🗑`.
- Реализуется через **дженерик-репозиторий** `Repository[T]` и **дженерик-вью** `CrudView[T]`, конфигурируемые описанием колонок и формы редактирования. Это и есть «переиспользование кода» из требований.
- Пагинация — серверная (`LIMIT/OFFSET` + `COUNT(*)` отдельным запросом, либо keyset для крупных таблиц — допустимо `LIMIT/OFFSET` для простоты).
- Сортировка по любой колонке.

### 5.4 Произвольный SQL
- Экран «SQL-консоль» — доступен только при наличии права `sql.execute`.
- Выполнение в **отдельной транзакции** с явным `ROLLBACK` по умолчанию и кнопкой «Применить» (`COMMIT`).
- Read-only режим (по умолчанию): подключение под пользователем БД с правом только `SELECT`. Полный режим — под основным пользователем, помечен красным.
- Результат — таблица; ошибка — текстовое поле; история запросов (локально, JSON в каталоге пользователя).

### 5.5 Сервисный режим
Отдельный раздел в админ-панели (право `service.testdata`):
- «Очистить БД» — TRUNCATE всех доменных таблиц (системные не трогаем), с подтверждением.
- «Засеять демо-данными» — фикстура: ~5 секций, ~15 групп, ~50 туристов/спортсменов, ~10 тренеров, ~5 маршрутов, ~10 походов, ~30 тренировок. Скрипт идемпотентен.
- «Сбросить пароль пользователя» — генерирует временный.
- «Экспорт дампа БД» — кнопка вызывает `pg_dump` через subprocess, сохраняет `.sql` (если pg_dump доступен в PATH; иначе — внятная ошибка).

### 5.6 Темы и UI/UX
- Минимум две темы: **Light** и **Dark**, переключаются в шапке окна без перезапуска.
- Все цвета/радиусы/отступы — в **одном файле дизайн-токенов** (`theme/tokens_light.py`, `tokens_dark.py`), QSS генерируется из них через шаблон. Никаких хардкодных цветов в виджетах.
- Шрифт — системный, иконки — `qtawesome` (Font Awesome).
- Главное окно — sidebar-навигация (разделы: Справочники, Тренировки, Походы, Запросы, SQL, Администрирование). Видимость разделов — по правам.
- Все диалоги модальные, имеют валидацию формы с подсветкой ошибок.

## 6. Нефункциональные требования

### 6.1 Архитектура
Слои (сверху вниз):
```
ui/         — PySide6 виджеты, окна, темы
viewmodels/ — состояние экранов, обёртка над сервисами
services/   — бизнес-логика, проверки прав, транзакции
repositories/ — доступ к данным (SQLAlchemy)
models/     — ORM-модели
db/         — engine, session, миграции
core/       — конфиг, логирование, ошибки, контейнер
```
- Никаких прямых SQL-вызовов из UI.
- Сервисы получают зависимости через простой DI-контейнер (фабрики), не через глобалы.
- Принцип SOLID — особенно SRP и DIP (UI ↔ Service через интерфейсы там, где это даёт упрощение тестов).

### 6.2 Целостность данных (на уровне БД)
- Все FK обязательны и заданы (`ON DELETE` решается осознанно: `RESTRICT` по умолчанию, `CASCADE` только для слабых сущностей).
- `CHECK`-ограничения: пол (`M/F`), даты (рождение < сегодня), длина маршрута > 0, дней > 0, количество участников похода 5–15, ставка ≥ 0, kind походов и маршрутов — `CHECK IN (...)`.
- `UNIQUE` на естественные ключи (логин пользователя, код права, имя роли, (route_id, point_order)).
- **Триггеры:**
  - При вставке `TripParticipant` с ролью `instructor`: проверять, что человек уже ходил в поход не меньшей сложности.
  - При закрытии похода: обновлять `Tourist.max_passed_difficulty_id`, если новый поход был сложнее.
  - При смене типа `unplanned → planned`: запрет, если не задан `TripPlan` на каждый день.
  - Проверка типа похода vs специализация туриста (на сплав — только умеющие плавать; для простоты — флаг `can_swim` у Tourist).

### 6.3 Кодстайл
- `from __future__ import annotations` — в каждом `.py`.
- Модульный docstring в начале каждого файла (1–3 строки: что делает модуль).
- Docstring **у всех публичных** функций, методов, классов. У приватных (`_name`) — по необходимости.
- Type hints везде. `Any` — только осознанно.
- Ruff: правила `E,F,I,B,UP,SIM,RUF,D` (pydocstyle — для публичного API), `line-length=100`.
- Pre-commit хук: `ruff check --fix`, `ruff format`.

### 6.4 Конфигурация
`.env.example` коммитится; реальный `.env` — в `.gitignore`. Ключи:
```
DB_HOST, DB_PORT, DB_NAME, DB_USER, DB_PASSWORD, DB_READONLY_USER, DB_READONLY_PASSWORD,
APP_LOG_LEVEL, APP_THEME (light|dark), APP_DEFAULT_ADMIN_PASSWORD
```

### 6.5 Логирование
- Один логгер на корне, конфиг в `core/logging.py`.
- Файл `logs/app.log` (ротация по 5 МБ × 5).
- Уровень из `.env`. Все исключения сервисного слоя логируются с трейсбеком.

### 6.6 Тесты
Минимально:
- unit-тесты на сервисы (с in-memory SQLite или test-схемой Postgres);
- unit-тесты на ACL (`require` срабатывает корректно);
- smoke-тест запуска UI через `pytest-qt`.

## 7. Развёртывание (целевой UX)

```
git clone <repo>
cp .env.example .env          # отредактировать пароли
docker compose up -d           # Postgres + Adminer
uv sync                        # установка зависимостей
uv run alembic upgrade head    # миграции
uv run python -m app.seed_admin   # создание суперадмина
uv run python -m app           # запуск GUI
```
- Adminer на `http://localhost:8080` — для просмотра БД.
- README содержит этот чек-лист, скриншоты главных экранов, описание ролей по умолчанию.

## 8. Что в проект НЕ входит

Чтобы не разрастаться:
- Веб-интерфейс, мобильное приложение, REST API.
- Полноценный i18n (UI на русском, без переключения языка).
- Сетевая многопользовательская блокировка записей (optimistic locking через `version` — опционально, не обязательно).
- Печатные отчёты (PDF). Экспорт — только CSV.
- Резервное копирование, кроме ручного `pg_dump` из сервисного раздела.
